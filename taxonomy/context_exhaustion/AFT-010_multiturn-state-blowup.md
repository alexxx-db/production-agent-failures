## [AFT-010] Multi-turn State Blowup

**Class:** Context Budget
**Severity:** P1
**Stacks Affected:** OpenClaw/Ari, LangGraph, Databricks Mosaic AI Supervisor
**First Observed:** Long-running WhatsApp user sessions in OpenClaw/Ari exceeding 50+ turns

---

### What I Expected

The agent maintains conversation coherence across a multi-turn session. When the context window approaches capacity, the system gracefully manages history — either summarizing or intelligently trimming — while preserving the user's core preferences and the system prompt.

### What Actually Happened

At approximately turn 40 in a WhatsApp conversation, the agent began violating its own system prompt instructions. It started responding in English to a user who had established Spanish as their preferred language in turn 2. It forgot dietary preferences established in turn 5 that it had correctly used in turns 6-35. The system prompt — which was at the beginning of the message history — had been silently truncated by the API's automatic context management.

No error was raised. No log line indicated truncation. The model's response confidence scores remained identical. The only signal was a user complaint: "Why are you suddenly speaking English?"

```
Turn 1-35:  system_prompt (800 tokens) + history (28,000 tokens) = 28,800 tokens
Turn 36-40: system_prompt (TRUNCATED) + history (31,200 tokens) = 31,200 tokens
                          ^^^^^^^^^^^
                          Model now operating without core instructions
```

### Why It Was Non-Obvious

The Anthropic and OpenAI APIs do not raise errors when context is truncated — they silently truncate from the beginning of the message array. The system prompt is the first message, so it is the first to be dropped. The agent continues responding, and its responses are still fluent and contextually relevant to the most recent turns. The quality degradation is subtle — it manifests as instruction drift, not as gibberish.

### First (Wrong) Mitigation

Implemented naive history trimming: when token count exceeds 80% of context window, drop the oldest user/assistant message pairs from the history. This preserved the system prompt but destroyed early-session context that contained user preferences, established facts, and conversation-level state. The agent would "forget" things the user told it 30 minutes ago while remembering the last 5 minutes perfectly.

### Root Cause

There is no context management strategy that works when applied at the point of exhaustion. By the time you're at 80-90% of context capacity, any trimming strategy loses information. The problem is not "how do you trim" — it's "when do you summarize." Summarization must happen proactively, well before the context window is pressured, so that the summary can be validated for completeness.

### Correct Mitigation

Context budget tracking with summarization checkpoints at 60% of context limit. At 60%, the system generates a structured summary of the conversation so far — user preferences, established facts, pending tasks, conversation tone — and replaces the full history with `[system_prompt, summary, last_N_turns]`. The summary is generated by the same model with a dedicated summarization prompt, not by simple truncation.

```python
class ContextBudgetManager:
    def __init__(self, max_tokens: int, summarize_at: float = 0.6):
        self.max_tokens = max_tokens
        self.threshold = int(max_tokens * summarize_at)

    def check_and_summarize(self, messages: list[dict]) -> list[dict]:
        current_tokens = count_tokens(messages)
        if current_tokens < self.threshold:
            return messages

        system_prompt = messages[0]
        recent_turns = messages[-6:]  # Keep last 3 exchanges
        history_to_summarize = messages[1:-6]

        summary = generate_summary(history_to_summarize)
        return [system_prompt, {"role": "assistant", "content": f"[Session summary: {summary}]"}] + recent_turns
```

The 60% threshold is not arbitrary — it leaves enough headroom for the summary generation itself to fit within the context window, plus 10-15 more turns before the next summarization checkpoint.

### Detection Signal

Track `session_token_count / max_context_tokens` as a metric per active session. Alert at 55%. If you see sessions regularly crossing 60% without summarization, you will hit this failure — it's a matter of when, not if.

### References

The Anthropic API documentation mentions context window limits but does not document the silent truncation behavior for messages that exceed it. OpenAI's documentation is similarly vague. This is a "known unknown" in the community — everyone knows context windows are finite, but the failure mode of silent instruction loss is not well-documented.
